Overview

An AI Beamtime

  • Staff complete a form for the upcoming experiment

    • Beam size
    • Sample environment
    • Duration
  • AI aligns the beamline according to configuration
  • User completes a form detailing their samples

Overview06
Motivation

Why LLM?

(my horse works fine)

  • Steer the experiment plan with natural language
  • Multimodal capability: send it images of plots or of the sample, and it can incorporate that data into its reasoning
  • Dynamic adaptability — the agent reasons about what it sees, and can troubleshoot issues as they arise.
  • Accelerate through the long tail of automation completeness, handling edge cases that would each require bespoke logic
  • Build ahead of the next model
Motivation07
Premise

How can we let AI run the beamline?

  • An LLM is a next-word-predictor — On its own, it cannot do anything
  • Tools are written to execute real actions based on what the LLM writes, and are fully configurable.
  • We have full granular control over what the LLM is allowed to do through choosing which tools to equip it with.
Premise08
Starting point

Simplest case

  • Give Claude Code full and direct access to SPEC, and to the SPEC macro definitions
  • Tell it to "follow this recipe to align the beamline"
  • It works — the frontier LLMs and harnesses available today are extremely capable

Downsides

Inefficient
In a poorly constrained system, the agent may wander through unproductive loops
Unconstrained capability
Agent has the capability to do anything — move upstream components, disable motor limits, etc

Takeaway

LLMs are capable. The accompanying software architecture does not need to be a hugely elaborate system of "making them good enough" — it's about setting up appropriate guardrails for efficiency and safety.

Starting point09
Approach

Guardrails — Equip an agent with specific tools

  • A tool here is a Python function that calls one SPEC command over TCP
  • Wrap all tools together into one CLI bash command: beamtimehero
  • No arbitrary code execution — the agent is given access to this one single command
  • Individual agents (beamline aligner, sample aligner, data collector) are given access to a relevant subset of the tool catalog
  • Construct a clear set of abilities and tasks — empower the agent to be adaptive and flexible within controlled constraints

Implementation

beamtimehero CLI

$ beamtimehero --help
tool Data queries, analysis, plots
db Plan, progress, steering queue
spec-read Motor positions, beam status
spec-write Motor moves, scans, energy
ref Procedure documents
steering Staff guidance queue
 
$ beamtimehero spec-write run-xas --help
--edge Absorption edge (e.g. Fe_K)
--count-time Seconds per point
--filters Number of Al filters
--justification (required)
 
$ beamtimehero spec-write run-xas \
--edge Fe_K --count-time 2 --filters 3 \
--justification "Rep 4 on S-A2, SNR 198"
 
[action_log] logged → id=1847
[spec] run_xas Fe_K ct=2 filt=3
[spec] scan #412 complete, 287 points
  • The agent gets access to one bash command: beamtimehero
  • Progressive discovery — agents call --help to learn what's available, then traverse the tree and drill in as needed.
  • 100+ subcommands — the entire beamline interface funnels through this single tool
  • Context-efficient, no 100-tool schema dump per request

Experimenter steering
The experimenter can send commands and guidance to the agent while it operates — skip samples, reorder the queue, adjust scan counts, extend time budgets. The agent acknowledges and re-plans around the change.
Roadblock escalation
If the agent hits a condition it can't resolve — hardware fault, unexpected cryostat behavior, ambiguous data — it pauses and messages staff on Slack. Collection resumes once staff responds.
